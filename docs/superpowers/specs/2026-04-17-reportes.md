## Context

FM CRM manages billing cycles, content consumption, and pipeline phases for media clients. The app uses:
- **Next.js** (App Router, Server Components by default), `export const dynamic = 'force-dynamic'` on all data pages
- **Supabase JS v2** (`@supabase/supabase-js`) — sequential queries required; no cross-table `.eq()` chaining
- **Tailwind CSS** with custom design tokens
- **TypeScript** throughout

## Feature A: Reporte por cliente

### Route
`/clients/[id]/report` — new file at `src/app/(app)/clients/[id]/report/page.tsx`

### Purpose
A printable/downloadable report for a single client showing the current billing cycle's consumption, pipeline status, and history.

### Access point
A "Ver reporte" link button is added to `ConsumptionPanel.tsx` in the action buttons row, next to "Editar cliente". It only renders when `cycle` exists (the component already receives `cycle` as a prop).

### Page sections

#### 1. Header
- Client name (large), plan name badge, client status badge
- Cycle period: `period_start` – `period_end` formatted as "1 ene – 31 ene 2026"
- Payment status: "Pagado" (green) or "Sin pago" (red)
- Days left: positive = days remaining, 0 = "Vence hoy", negative = "Vencido"
- Print-only: hide action buttons via `@media print { .no-print { display: none } }`

#### 2. Consumo por tipo
Table with columns: Tipo de contenido | Consumido | Límite | Restante | % Usado  
Rows: only content types where `limits[type] > 0`  
Progress bar per row (green < 70%, amber 70–89%, red ≥ 90%)

#### 3. Estado del pipeline
Count of pieces per phase from the current cycle (non-voided, `PIPELINE_CONTENT_TYPES` only).  
Show only phases with count > 0.  
Display as a simple summary row: phase label + count badge.

#### 4. Historial de ciclos
Table of past cycles (`status IN ('archived', 'pending_renewal')`), ordered by `period_start DESC`.  
Columns: Período | Plan | Estado de pago  
Plans are looked up from a plans map (fetched separately).

#### 5. Action buttons (`.no-print`)
- "Descargar PDF" — triggers `window.print()`
- "Descargar CSV" — client component `<CsvDownloadButton>` receives rows as prop and generates blob download

### CSV format (Feature A)
Filename: `reporte-{clientName}-{month}-{year}.csv` (lowercase, spaces replaced with `-`)  
Columns: Tipo,Consumido,Límite,Restante,% Usado  
One row per active content type (limit > 0).

### Print CSS
Inline `<style>` tag inside the page component:
```css
@media print {
  .no-print { display: none !important; }
  body { background: white; }
  .glass-panel { box-shadow: none; border: 1px solid #dfe3e6; }
}
```

## Feature B: Reporte de gestión interna

### Route
`/reports` — new file at `src/app/(app)/reports/page.tsx`

### Purpose
Internal operational dashboard showing aggregate consumption, pipeline status, revenue, and a per-client list — all for active clients in their current cycle.

### Nav entry
Add "Reportes" link to `src/components/layout/Sidebar.tsx` in the `navItems` array, after "Pipeline" (index 2, before "Renovaciones").

### Page sections

#### 1. Tarjetas resumen
Four summary cards:
- **Clientes activos**: count of clients where `status = 'active'`
- **Clientes pausados**: count where `status = 'paused'`
- **Clientes morosos**: clients where `status = 'overdue'` OR (cycle exists, `period_end < today`, `payment_status = 'unpaid'`)
- **MRR cobrado este mes**: sum of `price_usd` from plans for cycles paid this calendar month (`payment_date` between first and last day of current month)
- **Ingresos pendientes**: sum of `price_usd` from plans for current cycles with `payment_status = 'unpaid'`

> MRR and pending revenue require joining cycle → plan via `plan_id_snapshot`. Since Supabase v2 forbids cross-table `.eq()`, fetch cycles first, then fetch plans by their IDs in a separate query.

#### 2. Producción total (este mes)
For all active clients with a current cycle: sum each content type consumed across all clients.  
Display as a horizontal table or grid of cards: type label | total consumed | sum of limits | aggregate %  
Only show types where at least one client has `limits[type] > 0`.

#### 3. Resumen de pipeline
Count of pipeline pieces (non-voided, `PIPELINE_CONTENT_TYPES`) per phase across all current cycles.  
Show all phases from `PHASES` (show 0 for empty phases — unlike Feature A which hides empty phases).

#### 4. Lista de clientes activos
Table: Cliente | Plan | Progreso | Estado pago | Días restantes  
- "Progreso" = total consumed / total limits (e.g., "23 / 80") with a thin progress bar
- "Días restantes" = colored number (red if ≤ 0, amber if ≤ 3, green otherwise)
- Only clients with status 'active' and a current cycle

#### 5. CSV download (`.no-print` button)
Filename: `reporte-gestion-{month}-{year}.csv`  
Columns: Cliente,Plan,Consumido Total,Límite Total,% Consumo,Estado Pago,Días Restantes  
One row per active client.

## Shared component: `CsvDownloadButton`

**File:** `src/components/reports/CsvDownloadButton.tsx`  
**Type:** `'use client'` Client Component  

Props:
```ts
interface CsvDownloadButtonProps {
  rows: string[][]      // each inner array is one CSV row of strings
  headers: string[]     // column headers
  filename: string      // e.g. "reporte-cliente-abril-2026.csv"
  label?: string        // button label, default "Descargar CSV"
}
```

Behavior: on click, builds a CSV string, creates a `Blob('text/csv;charset=utf-8;')`, creates a temporary `<a>` element with `window.URL.createObjectURL`, clicks it programmatically, then revokes the URL.

## Data fetching constraints

All Supabase fetches follow these rules (v2 SDK):
1. Never chain `.eq()` across joined tables
2. Use sequential `await` calls — fetch parent, extract IDs, then fetch children `.in('id', ids)`
3. Guard against empty arrays before `.in()` (Supabase v2 returns all rows if `ids` is empty)
4. All pages use `export const dynamic = 'force-dynamic'`
